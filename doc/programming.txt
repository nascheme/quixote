Quixote Programming Overview
============================

This document explains how a Quixote application is structured.  Read
demo.txt first to see a running example.  A Quixote application has three
parts:

1) A *root directory* -- an instance of ``quixote.directory.Directory``
   (usually a subclass).  URL traversal starts here; see AGENTS.md and
   ``quixote/directory.py`` for the ``_q_*`` traversal protocol.

2) A *Publisher* that wraps the root directory and turns HTTP requests into
   responses.  Optional configuration and a logger are passed to it.

3) The bulk of your *application code*, reached when the Publisher calls the
   root directory's ``_q_traverse()`` method with the request path.

The smallest possible application is ``quixote/demo/mini_demo.py``; the rest
of this document refers to it.


The Publisher
-------------

You create a ``Publisher`` by giving it a root directory, plus optional
configuration::

    from quixote.publish import Publisher

    def create_publisher():
        return Publisher(RootDirectory(), display_exceptions='plain')

The Publisher translates each request URL into a Python object and calls it
(a method, function, or PTL template) to produce the response body.  The only
required customization is the root directory.


Serving the application
-----------------------

The Publisher does not talk to the network itself; a server adapter or WSGI
server does.  There are three common ways to serve an application.

* **Built-in server, for development.**  The ``quixote.server`` package
  provides adapters (``simple_server``, ``threaded_server``, ``scgi_server``,
  ``wsgi_server``, ...).  Each exposes a ``run(create_publisher, host, port)``
  function that takes your zero-argument publisher factory::

      from quixote.server.simple_server import run
      run(create_publisher, host='localhost', port=8080)

  ``python3 -m quixote run`` uses ``simple_server`` this way to serve the
  demo.

* **WSGI, for deployment.**  ``quixote.get_wsgi_app()`` returns a WSGI
  application for the current publisher, so you can serve Quixote from any
  WSGI server.  See wsgi.txt.

* **SCGI behind a front-end web server, for production.**  Run
  ``quixote.server.scgi_server.run(create_publisher, port=...)`` as a
  long-running process and have a front-end server (e.g. Apache with
  ``SCGIMount``, or nginx) forward requests to it.  This avoids paying
  interpreter and application startup on every request.

Switching backends is just a matter of importing ``run`` from a different
server module; the publisher factory stays the same.


Configuration
-------------

The Publisher uses defaults from ``quixote/config.py``.  Read that file for
the full list of settings and their documentation -- do not edit it, since
your edits would be lost on upgrade.  To customize, pass settings as keyword
arguments to ``Publisher(...)`` (as ``mini_demo.py`` passes
``display_exceptions``), or build a ``Config`` instance and pass it as the
``config`` argument.

The settings you are most likely to change:

  ``error_email``
    e-mail address to which error reports are sent
  ``error_log``
    file to which errors and tracebacks are written
  ``display_exceptions``
    set to ``'plain'`` (or ``'html'``) during development to render
    tracebacks in the browser; the default is off, favoring security over
    convenience


Logging
-------

The Publisher also accepts an optional ``logger`` argument; if omitted it
uses a ``DefaultLogger`` (see ``quixote/logger.py``).  Even with the default
logger you can control behavior through the ``access_log``, ``error_log``,
and ``error_email`` settings.

Quixote writes one line to the access log per request; split up for
readability, it looks like::

    127.0.0.1 - 2001-10-15 09:48:43
      2504 "GET /catalog/ HTTP/1.1"
      200 'Opera/6.0 (Linux; U)' 0.100s

The fields are: client IP address; current user (per the session manager, or
``-`` if none authenticates); request time in local time; process ID; the
HTTP request line; response status code; the ``User-Agent`` header; and the
time taken.  If ``access_log`` is ``None``, no access logging is done.

The error log receives application output to ``sys.stdout`` / ``sys.stderr``
and application tracebacks.  If ``error_log`` is not set, that output goes to
the stderr provided by the serving environment.  Redirecting stdout to the
error log is handy for debugging: ``print`` calls in your application show up
there.


Application code
----------------

This is the bulk of the application, and thanks to Quixote's design it is
ordinary Python -- there are no new hoops to jump through.  You may
optionally use PTL, which is Python with a different way of producing
function return values; see PTL.txt.

The Publisher constructs a request, splits the path into components, and
calls the root directory's ``_q_traverse()`` with that list.  The traversal
either returns a value that becomes the ``HTTPResponse`` body, or raises an
exception.  Exceptions are caught by the Publisher and handled according to
configuration and whether the exception is a ``PublishError`` (see
``quixote/errors.py``).
